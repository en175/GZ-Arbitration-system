请严格遵循以下开发原则：

一、技术与工程要求

1. 使用 Vue 3 + Vite 开发。
2. 适合中大型 Demo 项目持续迭代，不能用一坨代码堆在单文件里。
3. 必须按模块拆分，页面、组件、数据、样式、路由要有清晰边界。
4. 但考虑到我不懂开发、看不懂复杂工程，请尽量让“业务页面代码”集中在一个清晰易找的目录里，不要分散得太碎。
5. 我希望未来查阅时能快速定位，所以目录命名、页面命名、组件命名都要直观、中文业务语义清晰。
6. 除非必要，不要引入过重、过杂的技术栈；优先保证我后续能继续迭代和交接。
7. 优先使用前端 mock 数据，不接真实后端。
8. 页面之间需要具备演示级跳转、状态切换、假数据驱动能力。
9. 页面结构和样式实现不要过度依赖组件库默认长相，不要让我后续一改就被组件能力限制住。
10. 可以使用成熟 UI 组件库辅助开发，但必须以“可自由定制、可重塑样式、可持续迭代”为前提，不能做成明显的套壳组件风。
11. 后续如果需要扩展新模块、新页面、新演示链路，工程结构要能自然承接，而不是推翻重来。

二、目录结构要求
请参考“适合大型 Vue Demo 项目持续迭代”的思路来组织工程，但不要照搬任何现有项目业务。
我希望目录尽量清晰，推荐接近下面这种思路：

- src/views：主要业务页面，尽量把我最常看的页面都放这里，方便我查找
- src/views/某业务模块：该模块下的页面、局部组件、mock 数据、配置尽量放在一起
- src/components：通用组件
- src/router：路由
- src/styles：全局主题、变量、公共样式
- src/data 或 mock：演示数据
- src/services：少量通用服务

总原则：
“页面相关的东西尽量靠近页面放，通用的东西再抽出去”，避免我一个产品经理到处找代码。

三、UI 与视觉风格要求

1. 整体 UI 风格要高级、现代化、灵动、大气、前卫，但不能和法律/仲裁/政企场景太跳脱。
2. 不能做成死板传统政务风。
3. 要体现：
   - 专业感
   - 秩序感
   - 科技感
   - 可信赖感
   - 适度未来感
4. 推荐关键词：
   - 企业蓝 / 深蓝灰 / 冷白
   - 玻璃感/渐变感可少量使用
   - 卡片层次分明
   - 微动效自然克制
   - 阴影轻、圆角高级、边界清晰
   - 数据感、工作台感、智能化感
5. AI 能力区要突出“智能辅助、效率提升、风险提示、分析建议”，而不是做成花哨聊天机器人界面。
6. 设计上可以现代，但要让人一眼觉得仍然属于“法律 / 仲裁 / 专业机构”的系统。
7. 严格禁止使用暗黑模式。
8. 严格禁止使用 emoji 作为 icon。
9. 所有图标必须使用正式、统一、专业的图标体系，不允许用表情符号替代。
10. 整体 UI 要灵动大气，但不能浮夸，不能为了“未来感”牺牲业务理解和专业感。
11. 页面气质要让办案秘书觉得：专业、清晰、可信、顺手，而不是陌生、花哨、看不懂。

四、业务与演示要求

1. 这个 Demo 首先是给业务人员（尤其是办案秘书）看的，所以必须优先考虑他们的认知习惯和关注重点。
2. 页面表达要优先体现：
   - 我现在要处理什么
   - 哪些信息最重要
   - 当前该做什么动作
   - AI 帮了我什么
3. Demo 要突出“核心链路”和“AI 价值”，不要一上来铺满全系统。
4. 页面要适合汇报、演示、讲解，不只是能看，还要“讲得顺”。
5. 每个页面都要优先考虑：
   - 页面目标是什么
   - 给谁看
   - 核心信息是什么
   - 哪个区域最该突出
   - 哪些内容只是辅助展示
6. 尽量减少无意义复杂交互，优先保证演示清晰。
7. 所有页面先使用 mock 数据，数据要像真实业务数据，不要太假。
8. “立案”相关链路必须考虑支持“批量立案”的展示与流程表达，不能只按单条立案设计。
9. 如果涉及立案工作台、立案列表、立案审核、材料处理等页面，要优先考虑批量处理场景、批量筛选、批量操作、批量导入或批量确认的设计表达。
10. AI 的价值表达要贴近业务：例如材料识别、信息提取、风险提示、补正建议、优先级判断、批量辅助处理，而不是空泛地写“AI赋能”。
